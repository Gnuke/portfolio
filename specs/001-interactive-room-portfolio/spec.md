# Feature Specification: Only-One Room — 인터랙티브 방 포트폴리오 (MVP)

**Feature Branch**: `001-interactive-room-portfolio`

**Created**: 2026-07-13

**Status**: Draft

**Input**: User description: "진행해줘" — 기존 PRD(`prd.md`, v1.0 MVP)를 기반으로 한 베이스라인 명세

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 노트북으로 대표 프로젝트 확인 (Priority: P1)

포트폴리오 방문자(채용 담당자, 동료 개발자)가 방 안의 노트북을 클릭하면,
시점이 노트북 앞으로 부드럽게 다가가며 개발자가 현재 진행 중인 대표
프로젝트(Melolist-v3)의 정보가 표시된다. 이는 포트폴리오의 존재 이유인
"프로젝트 보여주기"를 담당하는 핵심 인터랙션이다.

**Why this priority**: 포트폴리오의 1차 목적은 방문자에게 대표 프로젝트를
보여주는 것이다. 이 스토리 하나만 구현되어도 "방을 탐험해 프로젝트를
발견한다"는 제품의 핵심 가치가 성립한다.

**Independent Test**: 방 화면에서 노트북을 클릭하여 시점 이동과 대표
프로젝트 정보 표시를 확인하고, 방으로 복귀할 수 있으면 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 방 전체가 보이는 메인 화면, **When** 방문자가 노트북을 클릭,
   **Then** 시점이 노트북 앞으로 이동하는 연출과 함께 대표 프로젝트
   (Melolist-v3) 정보 패널이 표시된다.
2. **Given** 노트북 정보 패널이 열린 상태, **When** 방문자가 "방으로
   돌아가기" 수단(복귀 버튼 또는 빈 벽 클릭)을 사용, **Then** 패널이 닫히고
   시점이 방 전체 보기로 복귀한다.
3. **Given** 방 전체가 보이는 메인 화면, **When** 방문자가 노트북 위에
   마우스를 올림(Hover), **Then** 노트북이 살짝 확대되고 밝아지며 그림자가
   강조되어 클릭 가능함을 알 수 있다.
4. **Given** 노트북 정보 패널이 열린 상태, **When** 패널이 표시되는 동안,
   **Then** 배경의 방이 패널 뒤로 계속 비쳐 보인다.

---

### User Story 2 - 책장으로 기술 스택 열람 (Priority: P2)

방문자가 책장을 클릭하면 개발자가 사용할 수 있는 기술 스택 목록
(예: Java, Spring Boot, React, Supabase, Docker, Git, AWS 등)이 정보 패널에
표시된다.

**Why this priority**: 프로젝트 다음으로 방문자(특히 채용 담당자)가 궁금해할
정보가 기술 역량이다. 대표 프로젝트와 독립적으로 가치를 전달한다.

**Independent Test**: 방 화면에서 책장을 클릭하여 기술 스택 목록이 패널에
표시되는지 확인하면 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 방 전체가 보이는 메인 화면, **When** 방문자가 책장을 클릭,
   **Then** 사용 가능한 기술 스택 목록이 정보 패널에 표시된다.
2. **Given** 책장 정보 패널이 열린 상태, **When** 방문자가 다른 오브젝트
   (예: 노트북)를 클릭, **Then** 책장 패널이 닫히고 해당 오브젝트의 패널로
   전환된다.

---

### User Story 3 - 화이트보드로 개발 예정 프로젝트 확인 (Priority: P3)

방문자가 화이트보드를 클릭하면 앞으로 개발 예정인 프로젝트 목록
(예: AI Room Portfolio, Mobile Application, Spring AI 등)이 표시된다.
MVP에서는 더미 데이터를 사용하며, 미구현 상태임이 자연스럽게 전달되도록
"Work in Progress" 형태의 안내를 함께 제공한다.

**Why this priority**: 개발자의 향후 계획과 성장 방향을 보여주는 보조
콘텐츠로, 핵심 콘텐츠(프로젝트·기술 스택)를 보완한다.

**Independent Test**: 방 화면에서 화이트보드를 클릭하여 예정 프로젝트
목록과 개발 중 안내가 표시되는지 확인하면 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 방 전체가 보이는 메인 화면, **When** 방문자가 화이트보드를 클릭,
   **Then** 개발 예정 프로젝트 목록이 정보 패널에 표시된다.
2. **Given** 화이트보드 패널이 열린 상태, **When** 방문자가 내용을 확인,
   **Then** 빈 화면이 아니라 "Work in Progress" / "Under Development" 형태의
   안내와 함께 예정 프로젝트가 표시되어 개발 중임을 이해할 수 있다.

---

### User Story 4 - 벽 스위치로 테마 전환 (Priority: P4)

방문자가 벽 스위치를 클릭하면 방 전체가 Light Mode와 Dark Mode 사이를
전환한다. 조명, 창문 밖 풍경, 블라인드, 그림자 등 방의 전체 분위기가 함께
변한다.

**Why this priority**: 콘텐츠 전달보다는 탐험의 재미와 몰입감을 높이는 경험
요소이다. 콘텐츠 스토리(P1~P3)가 성립한 뒤에 가치가 커진다.

**Independent Test**: 벽 스위치를 클릭하여 방 전체(조명·창밖·그림자 포함)의
분위기가 반대 테마로 전환되는지, 다시 클릭하면 원래 테마로 돌아오는지
확인하면 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** Light Mode의 방, **When** 방문자가 벽 스위치를 클릭, **Then**
   방 전체가 Dark Mode로 전환되고 조명·창밖·그림자 등 분위기가 함께 변한다.
2. **Given** Dark Mode의 방, **When** 방문자가 벽 스위치를 다시 클릭,
   **Then** 방 전체가 Light Mode로 복귀한다.
3. **Given** 임의의 테마 상태, **When** 방문자가 창문을 확인, **Then** 창문
   (배경)이 현재 테마를 반영한 모습으로 표시된다.

---

### User Story 5 - 첫 방문 인트로 연출 (Priority: P5)

방문자가 홈페이지에 최초 접속하면 짧은 인트로가 재생된다: 검은 화면 →
현관문이 열림 → 시점이 방 안으로 이동 → 조명이 켜짐 → 환영 문구 → 메인
화면(방 전체). 인트로는 2~4초 안에 끝나고 메인 화면으로 전환된다.

**Why this priority**: 첫인상을 만드는 연출 요소이지만, 인트로 없이도 방
탐험과 콘텐츠 열람(P1~P4)은 온전히 동작한다.

**Independent Test**: 페이지에 새로 접속하여 인트로 연출이 순서대로
재생되고 2~4초 내에 메인 화면으로 자동 전환되는지 확인하면 독립적으로
검증된다.

**Acceptance Scenarios**:

1. **Given** 방문자가 홈페이지에 접속, **When** 페이지가 열림, **Then**
   검은 화면 → 문 열림 → 방 진입 → 조명 점등 → 환영 문구 순서의 인트로가
   재생된다.
2. **Given** 인트로가 재생 중, **When** 재생이 끝남(2~4초), **Then** 메인
   화면(방 전체 보기)으로 자동 전환되고 모든 오브젝트와 상호작용할 수 있다.

---

### Edge Cases

- 정보 패널이 열린 상태에서 다른 오브젝트를 클릭하면? → 기존 패널이 닫히고
  새 오브젝트 기준으로 시점 이동·패널 표시가 이어진다 (중첩 패널 없음).
- 이미 선택된 오브젝트를 다시 클릭하면? → 상태가 깨지지 않고 해당 패널이
  유지되거나 자연스럽게 복귀한다 (빈 화면·이중 연출 금지).
- 창문(배경 요소)을 클릭하면? → 아무 패널도 열리지 않는다. 창문은 클릭
  대상이 아니다.
- 빈 벽(오브젝트가 없는 영역)을 클릭하면? → 열린 패널이 닫히고 방 전체
  보기로 복귀한다.
- 시점 이동 연출 중에 다른 오브젝트를 연달아 클릭하면? → 마지막 클릭
  기준으로 일관된 상태에 도달한다 (연출 충돌로 인한 깨진 화면 없음).
- 테마 전환 직후 패널이 열려 있으면? → 패널과 방 모두 새 테마를 일관되게
  반영한다.
- 미구현(향후 확장) 기능 영역이 노출되면? → 빈 화면 대신 "Under
  Development" / "Work in Progress" 안내가 표시된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 최초 접속 시 검은 화면 → 문 열림 → 방 진입 → 조명 점등 →
  환영 문구 순서의 인트로를 2~4초 동안 재생한 뒤 메인 화면으로 자동 전환해야
  한다 (MUST).
- **FR-002**: 홈페이지 전체는 단일 화면(방 하나)으로 구성되어야 하며, 어떤
  인터랙션도 페이지 이동(라우팅·새로고침)을 발생시켜서는 안 된다 (MUST NOT).
- **FR-003**: 상단 Navigation Bar 등 전통적 메뉴 UI를 제공하지 않아야 한다
  (MUST NOT). 방 안의 오브젝트가 메뉴 역할을 대신한다.
- **FR-004**: 방에는 클릭 가능한 오브젝트 4개(노트북, 책장, 화이트보드,
  벽 스위치)와 클릭 불가 배경 요소(창문)가 배치되어야 한다 (MUST).
- **FR-005**: 클릭 가능한 오브젝트에 마우스를 올리면(Hover) 살짝 확대,
  밝기 증가, 그림자 강조의 시각 피드백을 제공해야 한다 (MUST).
- **FR-006**: 노트북 클릭 시 시점이 노트북 앞으로 이동하는 연출과 함께 현재
  진행 중인 대표 프로젝트 1개(Melolist-v3)의 정보를 표시해야 한다 (MUST).
- **FR-007**: 대표 프로젝트 데이터는 여러 개를 담을 수 있는 구조여야 하지만
  (확장 대비), 좌우 슬라이드 등 다중 프로젝트 전환 UI는 이번 범위에서
  구현하지 않는다 (MUST NOT — 구조만 설계).
- **FR-008**: 책장 클릭 시 사용 가능한 기술 스택 목록을 정보 패널에 표시해야
  한다 (MUST).
- **FR-009**: 화이트보드 클릭 시 개발 예정 프로젝트 목록(더미 데이터)을 정보
  패널에 표시해야 한다 (MUST).
- **FR-010**: 벽 스위치 클릭 시 방 전체 테마가 Light ↔ Dark로 전환되어야
  하며, 조명·창밖·블라인드·그림자 등 방의 분위기 요소가 함께 변해야 한다
  (MUST).
- **FR-011**: 정보 패널은 반투명(유리 느낌) 스타일에 둥근 모서리와 부드러운
  그림자를 갖추고, 부드럽게 나타나고 사라지는 전환 효과를 사용해야 하며,
  패널이 열려 있는 동안에도 배경 방이 계속 보여야 한다 (MUST).
- **FR-012**: 정보 패널이 열린 상태에서 방 전체 보기로 복귀하는 수단(복귀
  버튼 또는 빈 영역 클릭)을 제공해야 한다 (MUST).
- **FR-013**: 정보 패널이 열린 상태에서 다른 오브젝트를 클릭하면 기존 패널을
  닫고 새 오브젝트의 패널로 전환해야 한다 (MUST).
- **FR-014**: 아직 구현되지 않은 기능 영역은 빈 화면을 노출하지 않고
  "Under Development" 또는 "Work in Progress" 안내 UI를 표시해야 한다 (MUST).
- **FR-015**: 방은 위에서 약 15~25도 내려다보는 시점의 2.5D 공간감(가구의
  측면·두께, 바닥·벽의 원근감)으로 표현되어야 하며, 평면 아이콘 나열 방식을
  사용해서는 안 된다 (MUST NOT).

### Key Entities

- **방 오브젝트 (Room Object)**: 방 안에 배치된 상호작용 단위. 속성: 종류
  (노트북/책장/화이트보드/벽 스위치/창문), 방 안의 위치, 클릭 가능 여부,
  연결된 콘텐츠.
- **프로젝트 (Project)**: 개발자가 진행 중이거나 예정한 작업. 속성: 이름,
  설명, 상태(진행 중/개발 예정). 진행 중 프로젝트는 노트북에, 예정
  프로젝트는 화이트보드에 연결된다.
- **기술 스택 항목 (Tech Stack Item)**: 개발자가 사용 가능한 기술. 속성:
  이름(예: Java, Spring Boot, React). 책장에 연결된다.
- **테마 (Theme)**: 방 전체의 분위기 상태. 값: Light / Dark. 조명, 창밖,
  그림자 등 방의 모든 시각 요소에 영향을 준다.
- **방 주인 프로필 (Profile)**: 포트폴리오 주인(개발자)의 소개 정보.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 첫 접속부터 방 전체 화면 도달까지(인트로 포함) 4초를 넘지
  않는다.
- **SC-002**: 방문자는 어떤 콘텐츠(대표 프로젝트, 기술 스택, 예정 프로젝트)
  든 클릭 1회로 도달할 수 있다.
- **SC-003**: 첫 방문자가 별도 안내 없이 Hover 피드백만으로 클릭 가능한
  오브젝트 4개를 모두 식별할 수 있다.
- **SC-004**: 사이트 이용 전 과정(인트로, 탐험, 패널 열람, 테마 전환)에서
  페이지 이동·새로고침이 0회 발생한다.
- **SC-005**: 테마 전환 시 방 전체(조명·창밖·그림자 포함)가 1초 이내에 새
  테마로 일관되게 반영된다.
- **SC-006**: 어떤 인터랙션 경로에서도 빈 화면(콘텐츠 없는 화면)이 노출되지
  않는다 — 미구현 영역은 항상 개발 중 안내로 대체된다.
- **SC-007**: 정보 패널이 열린 모든 상태에서 배경 방이 시각적으로 확인
  가능하다 (완전 불투명 오버레이 0건).

## Assumptions

- 모든 콘텐츠(프로필, 프로젝트, 기술 스택, 예정 프로젝트)는 정적 데이터로
  관리하며, 백엔드·데이터베이스 연동은 범위 밖이다 (PRD 향후 확장 계획).
- 대상 사용 환경은 데스크톱 브라우저(마우스 인터랙션 중심)이다. 모바일
  최적화·터치 전용 UX는 MVP 범위 밖이다.
- 인트로는 페이지 접속(로드)마다 재생되며, 재생 시간이 2~4초로 짧아 별도
  스킵 기능은 요구되지 않는다.
- 프로젝트 슬라이드(노트북 내 좌우 전환), 다중 프로젝트 관리, 외부 데이터
  연동, 프로덕션 배포, 방 커스터마이징, 방 템플릿, AI 방 생성은 모두 이번
  범위에서 제외된 향후 확장 항목이다.
- 이 명세는 기존 PRD(v1.0 MVP)를 기반으로 한 베이스라인 명세이며, 코드베이스
  에 이미 구현된 부분이 있을 수 있다 — 구현 현황 대조는 `/speckit-converge`
  단계에서 수행한다.
- 키보드 내비게이션·스크린 리더 등 접근성 심화 대응은 MVP 범위 밖이다
  (기본적인 대체 텍스트 수준은 권장).
