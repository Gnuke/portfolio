## Clarifications

### Session 2026-07-13

- Q: 관리자 페이지에서 관리(CRUD)할 콘텐츠의 범위는? → A: 전부 — 프로젝트
  (진행 중/완성), 화이트보드의 예정 프로젝트, 책장의 기술 스택 모두 관리
  대상이며 방문자 화면의 콘텐츠 원천을 정적 데이터에서 관리 데이터로 완전히
  대체한다.
- Q: 노트북 패널에서 진행 중/완성 프로젝트 노출 방식은? → A: 통합 목록 —
  하나의 이전/다음 목록으로 통합하고, 진행 중 프로젝트를 앞에 배치하며 각
  프로젝트에 상태 배지(진행 중/완성)를 표시한다.
- Q: 데이터 저장소 접근 불가 시 방문자 화면 폴백은? → A: 마지막 성공 데이터
  캐시 — 마지막으로 성공한 로드 결과를 방문자 기기에 보관해 두었다가 장애 시
  그것을 표시하고, 캐시가 없으면(첫 방문 등) 기존 내장 정적 데이터로
  폴백한다. 방문자는 장애 상황에서도 가능한 한 최신에 가까운 콘텐츠를 본다.
  (2026-07-13 재확인: 최초 "정적 데이터 폴백"에서 본 답으로 변경)
- Q: 프로젝트 설명 필드의 서식은? → A: 마크다운 — 관리자는 설명을 마크다운
  문법으로 작성하고, 방문자 패널은 서식이 적용된 형태로 안전하게 렌더링한다.
- Q: 이미지가 여러 장인 프로젝트의 방문자 패널 노출 방식은? → A: 갤러리 —
  대표 이미지를 먼저 표시하고, 패널 안에서 이전/다음으로 나머지 이미지를
  넘겨볼 수 있다 (페이지 이동 없음).
- Q: 같은 상태 그룹 내 프로젝트 노출 순서는? → A: 수동 순서 + 최신순 보조 —
  관리자가 지정한 표시 순서를 우선하고, 순서 미지정 항목은 최신 등록순으로
  뒤에 배치한다 (노트북·화이트보드 목록 공통).
- Q: 관리자 세션 유지 정책은? → A: 제공자 기본 정책 — 데이터 저장소 제공자의
  기본 세션 정책을 따른다 (사용 중 자동 연장, 명시적 로그아웃 전까지 유지).
  별도의 만료 시간을 직접 구현하지 않는다.
- Q: 장애 시 캐시 데이터 표시에 안내를 붙일지? → A: 조용히 표시 — 별도 안내
  없이 정상 콘텐츠와 동일하게 보여준다 (몰입형 경험 유지, 명세 001의 단일
  화면 원칙과 일관).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 관리자 로그인과 접근 제어 (Priority: P1)

포트폴리오 주인(관리자)이 본인 계정으로 관리자 화면에 로그인한다. 관리자
화면은 방문자용 방 화면과 분리된 별도 영역이며, 로그인하지 않은 사람은 어떤
관리 기능에도 접근할 수 없다. 관리자 계정은 포트폴리오 주인 1명의 계정만
존재한다.

**Why this priority**: 콘텐츠 관리 기능 전체의 보안 전제 조건이다. 접근
제어 없이 CRUD를 먼저 열면 누구나 포트폴리오 데이터를 조작할 수 있으므로,
가장 먼저 성립해야 하는 스토리다.

**Independent Test**: 관리자 화면 진입 시 로그인 요구 → 올바른 자격 증명으로
로그인 성공 → 잘못된 자격 증명 거부 → 로그아웃 후 관리 기능 접근 차단을
확인하면 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 로그인하지 않은 상태, **When** 관리자 화면 주소로 접근,
   **Then** 관리 기능 대신 로그인 화면이 표시된다.
2. **Given** 로그인 화면, **When** 관리자가 올바른 계정 정보(이메일과
   비밀번호)를 입력, **Then** 로그인에 성공하고 관리자 화면으로 진입한다.
3. **Given** 로그인 화면, **When** 잘못된 계정 정보를 입력, **Then** 로그인이
   거부되고 어떤 계정 정보가 틀렸는지 특정하지 않는 일반적인 오류 메시지가
   표시된다.
4. **Given** 로그인한 관리자, **When** 로그아웃을 실행, **Then** 세션이
   종료되고 이후 관리 기능 접근 시 다시 로그인이 요구된다.
5. **Given** 방문자용 방 화면, **When** 방문자가 방을 탐험, **Then** 관리자
   화면으로 이어지는 메뉴·버튼 등 진입 요소가 방문자 UI에 노출되지 않는다.

### User Story 2 - 프로젝트 등록·조회·수정·삭제 (Priority: P2)

로그인한 관리자가 프로젝트를 등록하고, 목록으로 확인하고, 내용을 수정하고,
필요 없어진 항목을 삭제한다. 각 프로젝트는 "진행 중", "완성", "예정" 중
하나의 상태를 가지며 상태를 서로 전환할 수 있다. 진행 중·완성 프로젝트는
방문자 화면의 노트북에, 예정 프로젝트는 화이트보드에 노출된다. 관리자는
책장에 노출되는 기술 스택 목록도 같은 방식으로 등록·수정·삭제할 수 있다.

**Why this priority**: 이 기능의 존재 이유인 콘텐츠 관리의 핵심이다. 로그인
(P1)이 성립한 직후 가장 먼저 가치를 전달한다.

**Independent Test**: 관리자로 로그인한 상태에서 프로젝트 1건을
등록 → 목록 확인 → 수정 → 상태 전환 → 삭제의 전체 주기를 수행하면
독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 로그인한 관리자, **When** 새 프로젝트의 필수 정보(제목, 설명,
   상태)를 입력하고 저장, **Then** 프로젝트가 등록되고 목록에 나타난다.
2. **Given** 프로젝트 목록, **When** 관리자가 목록을 조회, **Then** 각
   프로젝트의 제목과 상태(진행 중/완성/예정)를 한눈에 구분할 수 있다.
3. **Given** 등록된 프로젝트, **When** 관리자가 내용을 수정하고 저장,
   **Then** 변경 사항이 반영되고 목록·상세에서 수정된 내용이 보인다.
4. **Given** 등록된 "진행 중" 프로젝트, **When** 관리자가 상태를 "완성"
   또는 "예정"으로 변경, **Then** 상태가 전환되어 표시된다.
5. **Given** 등록된 프로젝트, **When** 관리자가 삭제를 시도, **Then** 확인
   절차(실수 방지)를 거친 후에만 삭제되고 목록에서 사라진다.
6. **Given** 프로젝트 등록/수정 화면, **When** 필수 항목을 비우고 저장을
   시도, **Then** 저장이 거부되고 어떤 항목이 누락되었는지 안내된다.
7. **Given** 로그인한 관리자, **When** 기술 스택 항목을 추가·수정·삭제,
   **Then** 변경된 기술 스택 목록이 저장되고 목록에서 확인된다.

### User Story 3 - 프로젝트 화면(이미지) 업로드 (Priority: P3)

관리자가 프로젝트를 등록·수정할 때 프로젝트 화면(스크린샷) 이미지를
업로드한다. 업로드한 이미지는 미리보기로 확인할 수 있고, 교체하거나 삭제할
수 있다.

**Why this priority**: 텍스트만으로는 프로젝트를 충분히 보여줄 수 없어
포트폴리오 가치를 크게 높이지만, CRUD(P2)가 먼저 성립해야 의미가 있다.

**Independent Test**: 프로젝트 등록/수정 화면에서 이미지 파일을 업로드하여
미리보기 확인 → 허용되지 않는 파일 거부 확인 → 이미지 삭제까지 수행하면
독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 프로젝트 등록/수정 화면, **When** 관리자가 이미지 파일을 선택해
   업로드, **Then** 업로드가 완료되고 미리보기가 표시되며 저장 후 프로젝트에
   연결된다.
2. **Given** 프로젝트 등록/수정 화면, **When** 이미지가 아닌 파일이나 허용
   용량을 초과한 파일을 업로드 시도, **Then** 업로드가 거부되고 허용 형식과
   용량이 안내된다.
3. **Given** 이미지가 연결된 프로젝트, **When** 관리자가 특정 이미지를 삭제,
   **Then** 해당 이미지가 프로젝트에서 제거된다.
4. **Given** 이미지가 여러 장 연결된 프로젝트, **When** 관리자가 대표 이미지
   를 지정, **Then** 방문자 화면에서 해당 이미지가 우선 노출된다.
5. **Given** 업로드 진행 중, **When** 업로드가 실패(네트워크 등), **Then**
   실패가 명확히 안내되고 재시도할 수 있으며 프로젝트 데이터가 불완전한
   상태로 저장되지 않는다.

### User Story 4 - 방문자 방 화면에 관리 데이터 반영 (Priority: P4)

방문자가 방의 오브젝트를 클릭하면 관리자가 관리하는 데이터가 표시된다:
노트북은 진행 중·완성 프로젝트를, 화이트보드는 예정 프로젝트를, 책장은 기술
스택을 보여준다. 정적으로 하드코딩된 데이터 대신 관리자가 관리하는 데이터가
포트폴리오의 원천이 된다. 노트북의 프로젝트가 여러 개면 방문자는 패널 안에서
이전/다음 방식으로 프로젝트를 넘겨볼 수 있다.

**Why this priority**: 관리 기능(P1~P3)의 최종 목적이지만, 관리 기능이 모두
성립해야 반영할 데이터가 존재한다. 기존 정적 데이터로도 방문자 경험은 이미
동작하므로 마지막에 연결한다.

**Independent Test**: 관리자 화면에서 프로젝트를 등록한 뒤 방문자 화면을
새로 열어 노트북 클릭 시 등록한 프로젝트가 표시되는지 확인하면 독립적으로
검증된다.

**Acceptance Scenarios**:

1. **Given** 관리자가 프로젝트를 1건 이상 등록한 상태, **When** 방문자가
   방의 노트북을 클릭, **Then** 관리자가 등록한 프로젝트 정보(제목, 설명,
   상태, 이미지)가 정보 패널에 표시된다.
2. **Given** 프로젝트가 여러 건 등록된 상태, **When** 방문자가 패널에서
   이전/다음 전환을 사용, **Then** 페이지 이동 없이 패널 안에서 다른
   프로젝트로 전환된다.
3. **Given** 관리자가 프로젝트 내용을 수정한 상태, **When** 방문자가 화면을
   새로 열어 노트북을 클릭, **Then** 수정된 내용이 표시된다.
4. **Given** 데이터 저장소에 접근할 수 없는 상황, **When** 방문자가 노트북을
   클릭, **Then** 빈 화면 대신 마지막으로 성공한 로드의 캐시 데이터가
   표시되고, 캐시가 없으면 기존 내장 정적 데이터가 폴백 콘텐츠로 표시된다.
5. **Given** 등록된 프로젝트가 0건인 상태, **When** 방문자가 노트북을 클릭,
   **Then** 빈 화면 대신 "준비 중" 형태의 안내가 표시된다.
6. **Given** 관리자가 예정 프로젝트와 기술 스택을 등록한 상태, **When**
   방문자가 화이트보드 또는 책장을 클릭, **Then** 관리자가 등록한 예정
   프로젝트 목록·기술 스택 목록이 각각 표시된다.
7. **Given** 이미지가 여러 장 연결된 프로젝트를 보는 중, **When** 방문자가
   패널 내 이미지 이전/다음을 사용, **Then** 대표 이미지부터 시작해 연결된
   이미지 전부를 페이지 이동 없이 넘겨볼 수 있다.

### Edge Cases

- 로그인 시도가 반복 실패하면? → 계정 존재 여부를 노출하지 않는 동일한 오류
  메시지를 유지한다 (보안).
- 로그인 세션이 만료된 상태에서 저장을 시도하면? → 작성 내용 손실을 최소화
  하며 재로그인을 요구하고, 인증 없는 저장은 거부된다.
- 삭제한 프로젝트에 연결돼 있던 이미지는? → 방문자 화면에서 더 이상 노출되지
  않으며, 저장 공간에 방치되지 않도록 정리 대상으로 관리된다.
- 이미지 업로드 도중 네트워크가 끊기면? → 실패 안내 후 재시도 가능, 반쪽
  상태(텍스트만 저장·이미지 유실 등 불일치)가 생기지 않는다.
- 허용 형식이지만 손상된 이미지 파일이면? → 방문자 화면에서 깨진 이미지
  대신 대체 표시가 나타난다.
- 동일 제목의 프로젝트를 중복 등록하면? → 허용하되 목록에서 구분 가능해야
  한다 (고유 식별은 시스템이 관리).
- 방문자가 방을 보는 동안 관리자가 데이터를 수정하면? → 방문자 화면은 다음
  로드 시점부터 최신 데이터를 반영하면 충분하다 (실시간 동기화 불요).
- 첫 방문(캐시 없음)인데 저장소 장애가 겹치면? → 기존 내장 정적 데이터가
  최종 폴백으로 표시된다 (FR-018의 ② 단계).
- 예정 프로젝트나 기술 스택이 0건이면? → 화이트보드·책장 패널도 빈 화면
  대신 안내를 표시한다 (노트북과 동일 규칙).
- 관리자 화면 주소를 방문자가 직접 입력해 접근하면? → 로그인 화면만 노출
  되고 데이터·관리 기능은 일절 노출되지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

**인증 및 접근 제어**

- **FR-001**: 시스템은 방문자용 방 화면과 분리된 관리자 전용 화면을 제공해야
  한다 (MUST). 관리자 화면 진입 요소는 방문자 UI에 노출되지 않아야 한다.
- **FR-002**: 관리자는 본인 계정(이메일과 비밀번호)으로 로그인할 수 있어야
  하며, 로그인 없이는 어떤 관리 기능(조회 포함)도 사용할 수 없어야 한다
  (MUST).
- **FR-003**: 관리자 계정은 포트폴리오 주인 1명분만 운영하며, 일반 사용자의
  회원가입 기능은 제공하지 않아야 한다 (MUST NOT).
- **FR-004**: 로그인 실패 시 어떤 정보가 틀렸는지 특정하지 않는 일반적인
  오류 메시지를 표시해야 한다 (MUST).
- **FR-005**: 관리자는 로그아웃할 수 있어야 하며, 로그아웃 또는 세션 만료
  후의 관리 요청은 거부되어야 한다 (MUST).

**프로젝트 관리 (CRUD)**

- **FR-006**: 관리자는 프로젝트를 등록할 수 있어야 한다 (MUST). 필수 정보:
  제목, 설명, 상태(진행 중/완성/예정). 선택 정보: 사용 기술 목록, 관련 링크,
  표시 순서. 설명은 마크다운 서식을 지원해야 하며(MUST), 방문자 패널은 이를
  서식이 적용된 형태로 안전하게(스크립트 실행 불가) 렌더링해야 한다 (MUST).
- **FR-007**: 관리자는 등록된 프로젝트 전체를 목록으로 조회할 수 있어야
  하며, 목록에서 제목과 상태를 구분할 수 있어야 한다 (MUST).
- **FR-008**: 관리자는 프로젝트의 모든 정보를 수정할 수 있어야 한다 (MUST).
  상태(진행 중/완성/예정) 간 전환을 포함한다.
- **FR-009**: 관리자는 프로젝트를 삭제할 수 있어야 하며, 삭제 전 실수 방지를
  위한 확인 절차를 거쳐야 한다 (MUST).
- **FR-010**: 필수 항목이 누락된 저장 요청은 거부되어야 하며, 누락된 항목을
  구체적으로 안내해야 한다 (MUST).
- **FR-011**: 프로젝트 데이터는 영구 저장소에 보관되어 화면을 닫거나 다시
  열어도 유지되어야 한다 (MUST).
- **FR-020**: 관리자는 책장에 노출되는 기술 스택 항목(이름, 선택적으로 분류·
  표시 순서)을 등록·조회·수정·삭제할 수 있어야 한다 (MUST). 삭제 확인,
  필수 항목 검증, 영구 보관 규칙(FR-009~011)은 기술 스택에도 동일하게
  적용된다.

**이미지 업로드**

- **FR-012**: 관리자는 프로젝트에 화면(스크린샷) 이미지를 업로드할 수 있어야
  한다 (MUST). 프로젝트당 여러 장을 허용한다.
- **FR-013**: 업로드는 일반적인 이미지 형식(PNG, JPEG, WebP, GIF)만 허용해야
  하며, 파일당 용량 상한을 두어야 한다 (MUST). 허용되지 않는 파일은 거부하고
  허용 형식·용량을 안내해야 한다.
- **FR-014**: 업로드한 이미지는 저장 전 미리보기로 확인할 수 있어야 하고,
  개별 삭제와 대표 이미지 지정이 가능해야 한다 (MUST).
- **FR-015**: 업로드 실패 시 명확한 안내와 재시도 수단을 제공해야 하며,
  프로젝트 데이터가 불완전한 상태로 남지 않아야 한다 (MUST).

**방문자 화면 반영**

- **FR-016**: 방문자 화면의 콘텐츠 오브젝트는 관리자가 관리하는 데이터를
  표시해야 한다 (MUST): 노트북은 진행 중·완성 프로젝트를, 화이트보드는 예정
  프로젝트를, 책장은 기술 스택을 표시한다. 기존 정적 데이터를 관리 데이터가
  완전히 대체한다.
- **FR-017**: 노트북 패널은 진행 중·완성 프로젝트를 하나의 통합 목록으로
  제공해야 한다 (MUST): 진행 중 프로젝트가 앞에 오고, 각 프로젝트에 상태
  배지(진행 중/완성)가 표시되며, 여러 건이면 방문자가 이전/다음 방식으로
  전환할 수 있다. 이 과정에서 페이지 이동이 발생해서는 안 된다 (MUST NOT —
  기존 명세 001의 단일 화면 원칙 유지). 같은 상태 그룹 안에서는 관리자가
  지정한 표시 순서를 우선하고, 표시 순서가 없는 항목은 최신 등록순으로 그
  뒤에 배치해야 한다 (MUST). 이 정렬 규칙은 화이트보드의 예정 프로젝트
  목록에도 동일하게 적용된다.
- **FR-018**: 데이터 로드에 성공하면 그 결과를 방문자 기기에 캐시로 보관해야
  하며, 데이터 저장소 접근에 실패하면 방문자 화면(노트북·화이트보드·책장
  패널 모두)은 다음 순서로 폴백해야 한다 (MUST): ① 마지막 성공 로드의 캐시
  데이터 → ② 캐시가 없으면 기존 내장 정적 데이터. 저장소는 정상이지만
  콘텐츠가 0건인 경우에는 "준비 중" 형태의 안내를 표시한다. 어떤 경우에도
  빈 화면을 노출해서는 안 된다 (MUST NOT — 명세 001의 FR-014와 일관).
  캐시·정적 폴백 콘텐츠는 별도의 장애 안내 없이 정상 콘텐츠와 동일하게
  표시한다 (몰입형 경험 유지).
- **FR-019**: 관리자가 변경한 내용은 방문자 화면의 다음 로드 시점부터 반영
  되어야 한다 (MUST). 실시간 동기화는 요구하지 않는다.
- **FR-021**: 노트북 패널의 각 프로젝트는 연결된 이미지를 패널 내 갤러리로
  제공해야 한다 (MUST): 대표 이미지가 먼저 표시되고, 이미지가 여러 장이면
  방문자가 패널 안에서 이전/다음 방식으로 전부 넘겨볼 수 있다. 이 과정에서
  페이지 이동이 발생해서는 안 된다 (MUST NOT).

### Key Entities

- **관리자 계정 (Admin Account)**: 포트폴리오 주인의 로그인 자격. 속성:
  이메일, 인증 자격 증명. 단 1개만 존재한다.
- **프로젝트 (Project)**: 관리 대상 콘텐츠 단위. 속성: 제목, 설명(마크다운
  텍스트), 상태(진행 중/완성/예정), 사용 기술 목록, 관련 링크, 표시 순서,
  생성·수정 시각, 연결된 이미지들. 진행 중·완성 프로젝트는 방문자 화면의 노트북에, 예정
  프로젝트는 화이트보드에 표시된다.
- **프로젝트 이미지 (Project Image)**: 프로젝트에 연결된 화면(스크린샷)
  파일. 속성: 원본 파일, 소속 프로젝트, 표시 순서, 대표 여부.
- **기술 스택 항목 (Tech Stack Item)**: 책장에 표시되는 관리 대상 콘텐츠.
  속성: 이름, 분류(선택), 표시 순서. 관리자가 등록·수정·삭제한다.
- **관리자 세션 (Admin Session)**: 로그인 상태의 유효 기간. 만료 시 관리
  기능 접근이 차단된다. 유효 기간·자동 연장은 저장소 제공자의 기본 정책을
  따르며 별도 만료 로직을 구현하지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자는 로그인부터 이미지 1장을 포함한 새 프로젝트 등록까지
  5분 이내에 완료할 수 있다.
- **SC-002**: 올바르지 않은 자격 증명으로는 관리 기능 접근이 100% 차단된다
  (조회 포함).
- **SC-003**: 관리자가 등록·수정한 내용이 방문자 화면의 다음 로드에서 100%
  반영된다.
- **SC-004**: 허용되지 않는 파일(형식·용량 위반)의 업로드 시도는 100% 거부
  되고, 매번 이유가 안내된다.
- **SC-005**: 데이터 저장소 장애 또는 프로젝트 0건 상황에서도 방문자 화면에
  빈 화면이 노출되는 경우가 0건이다.
- **SC-006**: 프로젝트 등록·수정·삭제 각 작업은 5초 이내에 완료 여부
  피드백을 제공한다.
- **SC-007**: 방문자는 등록된 모든 프로젝트를 클릭 1회 + 패널 내 전환만으로
  열람할 수 있다 (페이지 이동 0회 — 명세 001의 SC-004 유지).

## Assumptions

- 사용자는 데이터 저장소로 Supabase를 사용하겠다고 명시했다. 이는 구현 기술
  선택이므로 본 명세는 기술 중립적으로 기술하고, 구체 설계(인증·저장·파일
  보관 방식)는 `/speckit-plan` 단계에서 확정한다.
- 관리자 계정은 사전에 1개만 발급(프로비저닝)하며, 관리자 화면에는 회원가입
  절차가 없다. 비밀번호 재설정은 저장소 제공 기능 또는 수동 절차로 갈음하고
  이번 범위에서 별도 UI를 만들지 않는다.
- 이미지 파일 용량 상한은 파일당 5MB, 프로젝트당 이미지 최대 10장을 기본값
  으로 한다 (plan 단계에서 조정 가능).
- 관리 대상은 방문자 화면의 콘텐츠 전부다 (Clarifications 세션에서 확정):
  노트북(진행 중·완성 프로젝트), 화이트보드(예정 프로젝트), 책장(기술 스택).
  예정 프로젝트는 별도 유형이 아니라 프로젝트의 "예정" 상태로 통합 관리한다.
- 명세 001에서 "구조만 설계"로 유보했던 노트북 패널 내 다중 프로젝트 전환
  (이전/다음)은 이번 기능으로 활성화된다. 다만 연출 수준의 슬라이드
  애니메이션 품질은 요구사항이 아니다.
- 관리자 화면은 별도 주소로 접근하는 독립 화면이며, 방문자 경험(단일 화면
  원칙, No Navigation Bar)은 방문자용 방 화면에만 적용된다 — 관리자 화면은
  이 원칙의 적용 대상이 아니다.
- 관리자 1명만 사용하므로 동시 편집 충돌 처리는 범위 밖이다.
- 방문자 수 규모는 개인 포트폴리오 수준(동시 접속 수십 명 이하)을 가정한다.
